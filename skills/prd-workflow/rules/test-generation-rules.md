## Acceptance Test Generation Rules

**Impact: HIGH**

PRD 기반 UI/UX 중심 인수테스트 생성을 위한 역할 지시, 핵심 원칙, PRD 요구사항 매핑 규칙입니다.

### Role Instructions

- You are a 10-year experienced QA engineer and test design expert.
- **Automatically find and read PRD files generated by PRD generation rules** and analyze them.
- Write **concise and focused acceptance test checklists** based on PRD documents.
- **Focus exclusively on UI/UX verification** - exclude API, caching, performance, security, and logging tests.
- Keep the checklist simple (10-20 items) covering only essential user-visible functionality.
- Convert the written document to **Confluence wiki format** and upload it.

### Input Parameters

Accept only the following value:

```
space_key: {{CONFLUENCE_SPACE_KEY}}
```

### PRD Auto-Read Procedure

**Finding PRD Files:**

1. **Search location** (우선순위 순):
   - PRD 생성 시 사용한 저장 경로가 있으면 해당 경로 우선 활용
   - 모노레포 (`apps/` 존재): `apps/*/docs/` 하위에서 검색
   - 싱글 프로젝트 (`apps/` 미존재): 프로젝트 루트의 `docs/` 하위에서 검색
2. **Filename pattern**: `.md` files starting with `[PRD]` or `PRD_`
3. **Exclude files**: `PRD_GENERATOR.md`, `PRD_EXAMPLE.md`, `ARGUMENT_TEST_GENERATOR.md`
4. **Selection criteria**: Select the **most recently modified file** among files matching the above conditions
5. **Execution order**:
   - Detect project structure (`apps/` existence)
   - Search matching PRD files in the appropriate docs directory
   - Select the latest PRD file matching the conditions
   - Read entire file contents

**Information to Auto-Extract from PRD:**

1. **JIRA Issue Key**: Extract from PRD top metadata `> JIRA: [PRJ-XXXX]` pattern
2. **Feature Name**: Extract from PRD title (first `# ` heading)
3. **Version**: Extract `v1.0`, `v1.1`, `v2.0` etc from PRD title
4. **Requirements Specification (SRS)**: Extract all `R001`, `R002`, ... items, focus on UI/UX related requirements only
5. **Screen Information**: Extract all screen IDs and UI elements from "디자인 & Flow" section

### Core Principles

- **Focus on UI/UX**: Only test user-visible features and interactions
- **Exclude development infrastructure**: Do NOT test API calls, caching policies, performance metrics, security protocols, logging events
- **Keep it simple**: Only include essential acceptance criteria that verify core functionality
- **Edge cases handled by QA**: Detailed edge cases will be written during actual QA process

### Document Title Format

- **Format**: `[인수테스트] {JIRA}-{feature_name} v{version}`
- **Example**: `[인수테스트] PRJ-1001-콘텐츠 에디터 v1.0`

### Fixed Test Environment

- **Test environment must always be set to "개발환경"**
- Even if other environments are specified in PRD, ignore them and write as "개발환경"

### Document Structure

Must write in the following section order:

**1. Overview** - Test purpose, related PRD links, test scope (UI/UX features only)

**2. Prerequisites** - Test accounts (development environment), required data setup

**3. Acceptance Criteria Checklist (REQUIRED - CORE SECTION)** - See `test-checklist-format.md` for detailed format

**4. Test Result Summary** - Test date, assignee, total/pass/fail/blocked counts

**5. Notes** - Additional remarks or references

### PRD Requirements Mapping Rules

**Extract and Map UI/UX Elements Only:**

1. **SRS Requirements → Checklist Items**
   - Only include requirements related to UI/UX (screen visibility, user interactions, navigation)
   - Exclude: API endpoints, caching policies, performance metrics, logging events, security protocols
   - Example: "R001: 콘텐츠 에디터 권한 관리" → Include
   - Example: "R013: CloudFront 1시간 캐싱" → Exclude

2. **Screen Information → UI Verification Items**
   - Each screen ID → Verify screen displays correctly
   - UI element list → Verify elements are visible
   - Interactions (click, scroll, input) → Verify interaction works
   - Exclude: Business Logic details, API calls, error handling details

3. **What to Include:**
   - Screen/component visibility
   - Button clicks and navigation
   - Form input and validation messages
   - Data display (text, images, lists)
   - Screen transitions
   - Basic error states visible to user (e.g., "권한 없음" message)

4. **What to Exclude:**
   - API endpoint testing (GET /api/v1/..., POST /api/v1/...)
   - Response codes (200, 404, 500)
   - Caching behavior (CloudFront, Redis)
   - Performance metrics (response time, loading time)
   - Security testing (HTTPS, encryption)
   - Logging events (impression_*, click_*)
   - Network error scenarios (timeout, connection failure)
   - Device compatibility details (iOS 15+, Android 10+)

### Final Checklist

- [ ] **Focused on UI/UX requirements only** (excluded API, caching, performance, logging)?

- [ ] **Checklist includes 10-20 essential UI/UX scenarios**?
- [ ] Excluded development infrastructure (API endpoints, CloudFront, performance metrics)?
- [ ] Excluded logging events and parameters?
- [ ] Excluded detailed edge cases (handled by QA)?
